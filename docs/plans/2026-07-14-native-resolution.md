# Native-resolution tiles: variable-depth pyramids + Worker overzoom — planning doc

_Written 2026-07-14. Point-in-time; the code is the source of truth._

This plan is written assuming #80 will be merged before this work starts.

## Goal

Every accepted source builds and publishes at its native resolution, and tiles exist only where data earns their depth. Today both halves are violated in one direction each: `metadata.json max_zoom` caps discard real survey resolution (CUDEM's native z15 held at 13, a quarter of S-102's files and all of uk_surfzone held at 14, the Swiss lakes' z15–16 held at 14), while the shared vector maxzoom (`contour_run.bundle_maxz`) mints top-zoom tiles planet-wide — upsampled z9 GEBCO tiled to z14 over the entire ocean because five regional sources reach that deep somewhere. The principle this plan implements: **a source's cap is its data, not our tiling budget; depth is spent where data exists, synthesized where it doesn't.** Caps stop being a precision knob (they remain an ops escape hatch, valueless by default).

Three pieces, in dependency order: **(1)** the Worker learns vector overzoom synthesis (the sibling of its existing raster synthesis), **(2)** the vector bundle becomes one variable-depth tippecanoe run producing a single sparse `vector.pmtiles`, **(3)** the `max_zoom` caps are deleted and the covering/mosaic/terrain build to native — with a measured decision on refining raster render depth from per-stem to per-tile.

## Part 1 — Worker vector overzoom synthesis

The raster half already exists (`worker/src/index.ts`: deepest-ancestor walk within a cell, cubic-B-spline synthesis, `overzoomGate` limiter, etag'd cache). Vector gets the sibling:

- On a vector tile miss, walk ancestors via PMTiles directory lookups (the directory _is_ the availability index; lookups are cached range reads). Pure JS (`@mapbox/vector-tile` + `pbf` + `vt-pbf` — Workers-compatible, no WASM), behind the same limiter pattern as the raster gate.
- **The synthesis is a port of `tippecanoe-overzoom`'s semantics, not a naive scale+clip.** Per child tile: resolve parent geometry — including the tool's multi-parent form, needed where a child's buffer must be fed from a neighbor that leafed at a different depth — then scale ×2^Δz, clip to the child extent plus buffer, simplify to the child's effective grid (drop collinear/duplicate vertices), and repair polygons after clipping (degenerate rings, winding) so the depare fill contract survives. State the buffer arithmetic as a tested invariant, not an assumption: a child's buffer need in parent units is `b/2^Δz` and should fit inside the parent's own baked buffer `b`, but fixtures must include edge-adjacent children at large Δz and neighbors leafed at differing depths — exactly where that reasoning breaks if it breaks.
- **Leaves served at their own zoom also pass through clean/simplify.** Variable-depth leaves are written at full detail with no simplification or polygon cleaning; tippecanoe's README makes the serving layer responsible for both. Serving them raw is a bytes-and-validity risk, not a shortcut.
- **Correctness oracle: `tippecanoe-overzoom`, compared semantically.** Feed identical inputs through the C++ tool and the TS port; compare per feature after decode — layer name, feature id, geometry, property values _and types_ (the bundle coerces several numeric fields via `-T`; a port that reproduces geometry but strings the numerics silently breaks style filters), and `rank` (it orders depare fills). Bytes will differ (encoder ordering); semantics may not. The depare partition contract (pairwise-disjoint fills meeting at seams) is the regression canary.
- **Authoritative serving maxzoom moves to the manifest.** The Worker today advertises the vector archive's header maxzoom; under variable depth that header records the deepest _baked leaf_, which can be shallower than the covering's max child_z (a deep region whose content fully fits leafs early). Advertising it would stop clients requesting the deeper tiles and push them onto their own overzoom instead of the Worker's cleaned synthesis path. Add `vector.max_zoom` to the release manifest, set from the covering; the Worker advertises that, and a test covers the case where the deepest intended zoom has no baked leaf.
- Ships before or with Part 2 (a sparse archive served without synthesis renders holes in stock clients). Independently testable now against a hand-sparsified fixture archive.

## Part 2 — one variable-depth tippecanoe run; `bundle_maxz` dies

- **One invocation, named layers.** `--generate-variable-depth-tile-pyramid` stops subdividing wherever a tile already carries its content at full detail, emitting a sparse pyramid; its README explicitly assumes a tile server that overzooms ancestors (Part 1). Replace the three per-layer tippecanoe runs + tile-join fold with a single run: `tippecanoe --generate-variable-depth-tile-pyramid -L contours:… -L soundings:… -L depare:…`. The vector bundle already runs as one pass on the build box, so this is a flag/flow change, not a topology change — and a single run is what makes variable depth seam-free (sharded variable depth is only safe when shards own whole tile subtrees; see rejected alternatives). This is not just simplification — it is _required_ for variable depth: separately-tiled layers leaf at different depths, and the join would produce deep tiles that exist but lack the shallower-leafing layer, the exact layer-vanishing bug `bundle_maxz` exists to prevent (MapLibre overzooms a missing tile, never a missing layer). One run makes the leaf decision per-tile across all layers jointly. Delete `bundle_maxz`, `_global_maxz`, the shared-maxz plumbing in `soundings_run.bundle` / `depare_run.bundle`, and the join step.
- **The joint run's real blocker: tile-budget strategies are global flags, not per-layer.** Contours use `--drop-densest-as-needed`; depare uses `--coalesce-smallest-as-needed` and must _never_ drop a polygon (a dropped band is a hole in the partition); `tippecanoe --help` confirms every as-needed strategy is invocation-global. Candidate configuration to prove with fixtures before committing: global `--coalesce-smallest-as-needed` (merges, never drops) with per-feature minzoom carrying all zoom gating, guarded by per-tile safety assertions in the bundle self-check — depare band-area sum invariant per tile (no polygon lost or displaced by coalescing), no contour absent at or above its minzoom, sounding count monotone in zoom. If no single-command configuration survives the assertions, the revised architecture is: per-layer variable-depth runs + tile-join + a build-time _completion pass_ that overzoom-fills any layer whose leaf is shallower than a joined tile, running `tippecanoe-overzoom` itself on the box — deterministic, reuses Part 1's exact semantics, and restores per-layer flag freedom at the cost of a fourth bundle step.
- **Zoom-gated content must become per-feature minzoom.** An overzoomed leaf is frozen at its own zoom's content, so anything tippecanoe filters _by_ `$zoom` breaks under variable depth. The contour tier filter (`PER_ZOOM_FILTER`, the `-j $zoom` bands) is the violator: replace it with `tippecanoe.minzoom` per contour feature (each curve's tier entry zoom), the mechanism soundings already use. That means contours join soundings on the GeoJSONSeq path at bundle time (FGB can't carry the tippecanoe extension) — or the FGBs gain a minzoom attribute and a small conversion shim; implementer's call. Style impact: none — tiles at or above a curve's tier zoom carry it exactly as today. **Depare goes through the same shim**: its zoom floor is currently the run-level `-Z MIN_ZOOM` (z6), which dies in a joint run whose soundings need `-Z 0` — every depare feature gains `tippecanoe.minzoom = 6`, with a regression asserting no depare bytes (and no rendering) below z6. Past the floor it's the easy layer: unfiltered, full partition at every zoom, and its interior ocean tiles are what leaf earliest — where the win lives.
- **Verify, don't assume, the leaf/minzoom interaction.** tippecanoe documents that tiles with point features below basezoom or with dynamically-dropped features keep subdividing; whether per-feature `minzoom` on _lines_ forces subdivision until the minzoom is reached must be confirmed with a fixture before committing (a leaf that freezes below a curve's minzoom would hide the curve at native+ zooms). If it doesn't hold, fallback: keep a floor via `--minimum-zoom`-style banding of the run, or bake curves in at all zooms and thin in the style (accepting low-zoom byte growth — measure).
- **Empty-state discipline carries over**: the current bundle deletes a stale `soundings.pmtiles` when inputs vanish; the single-run world keys one output on the union of all three layers' member names + modules + flags (the existing `keys.stage_key` shape), so the invalidate logic simplifies rather than grows.
- **Measurements that gate this part** (one `bbox` build over a mixed cell — Boston or the Elbe): where ocean tiles actually leaf once depare is the only deep-water content; leaf tile bytes vs today's simplified maxzoom tiles; total archive size vs the current dense z14 build; single-run wall clock + RSS on the box (the scheduler's budget must fit it — tippecanoe is the one stage that can't shard without reintroducing seams, so it must fit in box memory; if it can't, the fallback is cell-subtree shards + join, which is seam-safe _only_ because shards would own whole subtrees).
- **Publication contract change**: `vector.pmtiles` becomes sparse. The Worker path is transparent (Part 1). Direct-archive consumers need `pmtiles` JS ≥ 4.1 with `errorOnMissingTile: true` (protomaps/PMTiles#321; it rides MapLibre's error-path fallback, so it works but is a workaround) — a README/CHANGELOG note. First-class client support is an open upstream proposal (maplibre-gl-js#5618, style-spec#938); we posted the producer case on #5618 (2026-07-14), and if it lands, the flag requirement and the Worker synthesis both become optimizations. The npm style package needs no schema change (Part 2 keeps tile content identical at unchanged zooms), but the release notes must state the client requirement — degraded (blank deep tiles in old clients without the flag), not broken.

## Part 3 — delete the caps; build to native

- Remove `max_zoom` from every `sources/*/metadata.json` (the covering's cap path stays as the documented ops escape hatch). The covering derives child_z from native overzoom as it already does; the mosaic key sees the maxzoom change and re-keys affected tiles; the incremental build rebuilds exactly those cells. No manual state clearing.
- **The uncap is not live until the published catalogs refresh.** Production builds read each source's published `catalog.json` (`seascape:max_zoom`), not `metadata.json` directly — a planet build dispatched before the source registration re-runs still builds capped (benign: the next build after the refresh picks it up, but it wastes a build). The metadata edit does change the registration recipe hash, so `sources.yml` re-registers on its own; the operational sequence per source is: edit metadata → let (or make) `sources.yml` re-run registration → verify the republished catalog carries no cap → then dispatch the planet build.
- **Cost profile (why this is last)**: each +1 child_z is 4× mosaic pixels and 4× terrain render work in affected stems. CUDEM to z15 is the big one — the whole US coastal strip. The box absorbs compute (scheduler-budgeted, pay-per-hour); the planet-build wall clock and the mosaic store size are the numbers to watch. Do it in two steps if the first measurement is ugly: uncap the genuinely-surveyed sources first (noaa_s102, uk_surfzone, Swiss lakes — small footprints, real information), CUDEM/estuarine second.
- **Raster minting can stay per-stem-dense initially.** Terrain renders each stem to child_z; within a fine stem most tiles are upsample (an S-102 ribbon is ~10–20% of its macrotile). The Worker already walks to the deepest present ancestor within a cell, so a _sparse_ render — skip an output tile at depth d unless a source file with native maxzoom ≥ d intersects it (the covering CSV rows carry per-file maxzoom; the mosaic index carries per-tile `seascape:maxzoom`) — is a pure build-side optimization with no serving change. Implement it only if the Part 3 measurement shows the dense z15 rings dominating build time or archive size; otherwise skip it (YAGNI).
- Ordering within Part 3: sparse-mint decision _before_ the CUDEM uncap if measurements say it's needed, since CUDEM is what makes dense z15 expensive.

## Part 4 (follow-on) — one sparse `raster.pmtiles`

The raster analog of Part 2, packaging-only (tippecanoe plays no role in the raster path, so the "variable depth" rule must be stated by us): collapse planet + per-cell overlays into a single sparse archive whose tile presence follows the Part 3 sparse-mint rule — a tile exists at depth d only where a source with native maxzoom ≥ d intersects it (covering CSV per-file maxzoom / mosaic index `seascape:maxzoom`). Why bother: today's planet+cells layout is effectively Worker-only-consumable (a direct consumer would have to reimplement cell routing), which contradicts the publish-archives-directly goal Part 2 serves for vector — and raster is the _easier_ client story, since stock MapLibre already parent-falls-back on within-range raster 404s (the working case in maplibre#5692). The Worker switches from manifest-cell routing to ancestor directory-probing within the one archive (cached range reads; the manifest keeps max_zoom for TileJSON).

Two costs make this a follow-on, not part of the main line: **(a)** synthesized ≠ rendered for raster — the Worker's B-spline upsample is not the per-zoom mosaic render (zoom-dependent smooth + quantization), so where sparse minting makes synthesis the _permanent_ representation of pure-upsample areas, the contour/shading agreement needs an eyeball pass; **(b)** single-archive finalize serializes the concat and transiently doubles disk on the box — much weaker than when the split was invented (400 GB volume), but a number to measure, and the "new sources add cells, never bytes-per-cell" property is deliberately given up (it buys nothing on a single box).

## Rejected alternatives

- **Planet-z8 vector + per-cell vector overlays**: an earlier sketch from when the vector bundle was sharded across 6 h-capped hosted runners — cell-aligned shards were the mechanism that made sharded variable depth seam-safe. The single-box, single-pass bundle makes it dead weight. Kept as the documented fallback if a planet single run outgrows box memory.
- **Per-layer variable-depth runs + tile-join**: reintroduces layer-vanishing at every depth boundary; `bundle_maxz`'s reason for existing, now per-tile instead of global. Only a joint run leafs coherently.
- **Serving-side layer compositing** (each layer its own sparse tileset; Worker concatenates per-layer best tiles per request): optimal per-layer sparseness, but direct-archive consumers would get either three files or a joined file with the vanishing bug — and the single joint run gets ~all the benefit with one artifact.
- **Waiting for maplibre #5618**: upstream client support would make sparse archives portable without the Worker, but it's an open proposal with no implementation; the Worker synthesis is needed for stock clients regardless, and becomes an optimization if/when #5618 lands.
- **Keeping caps as a quality signal**: quality belongs to the coverage layer's `zoc`/provenance work, not to resolution. Capping a trusted source's grid is discarding data to express doubt; if a source isn't trustworthy at its native grid, the answer is not accepting it, or flagging it — not blurring it.

## Verification

1. Worker synthesis: TS overzoom output ≡ `tippecanoe-overzoom` compared semantically per feature (layer name, feature id, geometry, property values and types, `rank`) across line/point/polygon fixtures — including edge-adjacent children at large Δz, neighbors leafed at differing depths (the multi-parent case), and tile-edge polygons with buffer; depare partition stays pairwise-disjoint and seam-exact through synthesis and through the leaf clean/simplify pass.
2. A sparse fixture archive served through the Worker renders identically (screenshot diff) to today's dense archive in the stock preview at every zoom, including past the local leaf depth.
3. Variable-depth leaf semantics: a contour feature with `tippecanoe.minzoom = n` is present in every served tile at z ≥ n over its geometry — baked or synthesized (this is the test that gates the whole `-j` → minzoom rework).
4. Soundings density at native+ zooms unchanged vs today over a surveyed harbor (the finest field's minzoom == child_z contract survives the joint run).
5. Joint-run safety assertions hold on a mixed-cell build: per-tile depare band-area sum invariant (nothing dropped or displaced by the global coalesce strategy), no contour missing at or above its minzoom, no depare bytes below z6.
6. The manifest's `vector.max_zoom` governs serving: a region whose deepest intended zoom has no baked leaf still serves correct synthesized tiles at that zoom through the Worker, and the TileJSON advertises the covering's max child_z, not the archive header's.
7. Post-uncap: the ICW/Elbe/Boston transects show strictly more detail at z15 than the z14 build (contour vertex density, sounding count), and _unchanged_ rendering over GEBCO-only ocean at every zoom.
8. Archive/build metrics recorded per part: vector.pmtiles size, tile-count-by-zoom histogram, planet build wall clock, box peak RSS — the plan's claims (ocean leafs early; top zoom stops dominating) are measurable, so measure them.

## Rollout

Part 1 is a Worker-only deploy, inert until sparse archives exist. Part 2 ships tiles + Worker together (a sparse archive needs the synthesis live; old dense archives remain servable — the synthesis path only fires on misses, which dense archives don't produce). The npm style package is unchanged in schema but the release CHANGELOG documents the direct-consumer client requirement (`pmtiles` ≥ 4.1 + `errorOnMissingTile`). Part 3 rolls source-by-source, but each step is two-phase operationally: the registration republish (so the source's `catalog.json` drops the cap — it's what builds read) must complete before the planet build that uncaps it; the content keys handle the rest.
